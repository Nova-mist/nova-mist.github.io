Control Flow

Defer, Panic and Recover

Defer used to delay execution of a statement until function exits.

Deferring doesn’t move to the end of the main function, and actually move it after the main function but before the main function return.

后进先出的执行原则，经常用于关闭资源。

func main() {
  fmt.Println("a")
  defer fmt.Println("b")
  defer fmt.Println("c")
}
package main

import (
	"fmt"
	"io/ioutil"
	"log"
	"net/http"
)

func main() {
	res, err := http.Get("http://www.baidu.com/robots.txt")
	if err != nil {
		log.Fatal(err)
	}
  // 关闭的步骤可以紧挨着开启资源的步骤
	defer res.Body.Close()
	robots, err := ioutil.ReadAll(res.Body)
	if err != nil {
		log.Fatal(err)
	}
	fmt.Printf("%s", robots)
}

defer 的本质是调用了方法但推迟执行。

下面的情况：Arguments evaluated at time defer is executed, not at time of called function execution.

a := "start"
defer fmt.Println(a)
a = "end"

Recover

Panic don’t have to be fatal.

Panic happen after deferred statements are executed. 意味着使用defer关闭资源的步骤在发生panic之前就完成了。

fmt.Println("start")
defer fmt.Println("this was deferred")
panic("something bad happened.")
fmt.Println("end ")
// defer cover与匿名方法
defer func() {
  // recover() 返回nil 如果程序没有正在panic
  if err := recover(); err != nil {
    // is panicing
    log.Println("Error:", err)
  }
}() // 调用方法
panic("something bad happened.")
fmt.Println("end ")

defer statement doesn’t take a function itself, but a function call.

func main() {
  fmt.Println("start")
  panicker()
  fmt.Println("end")
}

如果在调用的panicker方法中产生了panic，①没有处理panic整个程序就会中止，②但如果在方法中使用了recover处理panic那么只是panicker方法不再执行，整个程序还会继续运行，③但如果在recover无法处理panic，也可以重新发出panic，这时候整个程序就会中止。

Panic Occur when program cannot continue at all

  • Don’t use when file can’t be opened, unless it is critical
  • Use for unrecoverable events - cannot obtain TCP port for web server

race conditions

通常在主线程sleep之前，go调度器不会切换到别的线程。

WaitGroup synchronize(同步) multiple go routines together.

var wg = sync.WaitGroup{}
func main() {
  var msg = "Hello"
  wg.Add(1)
  go func(msg string) {
    fmt.Pringln(msg)
    wg.Done()
  }(msg)
  msg = "Goodbye"
  wg.Wait()
}

go routines are actually racing against each other. 竞争

a mutex is basically a lock 线程锁

读写锁，多线程可读取，单线程可写。

package main

import (
	"fmt"
	"runtime"
	"sync"
)

var wg sync.WaitGroup = sync.WaitGroup{}
var counter int = 0
var m = sync.RWMutex{}

func main() {
	runtime.GOMAXPROCS(100)
	for i := 0; i < 10; i++ {
		wg.Add(2)
		m.RLock()
		go sayHello()
		m.Lock()
		go increment()
	}
	// don't forget
	wg.Wait()
}

func sayHello() {
	// use printf with %v not println
	fmt.Printf("Hello #%v\n", counter)
	m.RUnlock()
	wg.Done()
}

func increment() {
	counter++
	m.Unlock()
	wg.Done()
}

Channel are designed to synchronize data transmission between multiple go routines.

var wg = sync.WaitGroup{}
func main() {
  ch := make(chan int)
  wg.Add(2)
  // receiving go routine
  go func() {
    i := <- ch
    fmt.Println(i)
    wg.Done()
  }()
  // sending go routine
  go func() {
    i := 42
    ch <- i // pass a copy of the data
    i = 27
    wg.Done()
  }()
  wg.Wait()
}

既可以是接收方也可以是发送方，但很想限定只能是接收方或发送方

🟠Go routines is designed to handle channel data only in one direction.

Go routines被设计为处理单向的channel数据。

var wg = sync.WaitGroup{}
func main() {
  ch := make(chan int)
  wg.Add(2)
  // receiving go routine
  go func(ch <-chan int) {
    i := <- ch
    fmt.Println(i)
    wg.Done()
  }(ch)
  // sending go routine
  go func(ch chan<- int) {
    ch <- 42
    wg.Done()
  }(ch)
  wg.Wait()
}

**buffered channels

普通情况下发送方和接收方是一对一的，buffered channels 常见缓存来处理发送接受频率不同的情况，防止因为执行效率的不同而造成block。

var wg = sync.WaitGroup{}
func main() {
  ch := make(chan int, 50) // got an internal data store
  wg.Add(2)
  // receiving go routine
  go func(ch <-chan int) {
    i := <- ch
    fmt.Println(i)
    i = <- ch
    fmt.Println(i)
    wg.Done()
  }(ch)
  // sending go routine
  go func(ch chan<- int) {
    ch <- 42
    ch <- 27
    wg.Done()
  }(ch)
  wg.Wait()
}
  • 🟠除了从发送信息产生的panic信息，没办法检测到channel关闭，如果向关闭了地channel发送数据会产生panic，这时候需要使用在defer函数中使用recover来处理panic.
  • 使用 for range 来遍历channel，会一直监听channel中可能的新的信息，如果不关闭channel，就会造成死锁。
var wg = sync.WaitGroup{}
func main() {
  ch := make(chan int, 50) // got an internal data store
  wg.Add(2)
  // receiving go routine
  go func(ch <-chan int) {
    for i: = range ch {
      fmt.Println(i)
    }
    wg.Done()
  }(ch)
  // sending go routine
  go func(ch chan<- int) {
    ch <- 42
    ch <- 27
    // close the channel
    close(ch)
    // can't send data to closed channel
    // ch <- 33 ❌
    wg.Done()
  }(ch)
  wg.Wait()
}

在接收方检测channel关闭的方法

go func(ch <- chan int) {
  for {
    if i, ok := <- ch; ok {
      fmt.Println(i)
    } else {
      break
    }
  }
  wg.Done()
}()

channel无法显示关闭的情况

主进程执行结束，channel也不再运行，可以使用defer的函数来关闭channel. 不是一个好办法

type logEntry struct {

var logCh = make(chan logEntry, 50)
func main() {

  defer func() {

func logger() {
  for entry := range logCh {

Select statements

在go中，没有内容的struct不占用内存，用此创建的channel不会收发真实的数据，只传递收发的事实。可以理解为只传输了标头没有数据？？

整个for statement会在消息被接受之前保持block

如果使用了default，就不再是blocking SELECT statement.

在并行的情况下，会有两个channel同时接收到了信息，此时会产生冲突，但case的顺序对此没有任何影响。

var logCh = make(chan logEntry, 50)
// var doneCh = make(chan bool) // require a variable to be allocated
var doneCh = make(chan struct{}) // signal only channel

func main() {
  go logger()
  logCh <- logEntry{} 
  time.Sleep(100 * time.Millisecond)
  // 程序结束后发送关闭channel的信号
  doneCh <- struct{}{} // 定义了一个空的struct并实例化
}

func logger() {
  for {
    select {
    case entry := <- logCh:
      fmt.Printf("")
    case <- doneCh:
      break
    }
  }
}
