HF设计模式

发表于 2021-05-13 更新于 2021-05-27

读完 Head First Java , 接着看看设计模式，因为之前已经看过了《大话设计模式》有了基本的了解，希望这次阅读应该能轻松一点。☕

OO基础：

  • 抽象
  • 封装
  • 多态
  • 继承

OO原则：

  • 封装变化
  • 多用组合，少用继承
  • 针对接口编程，不针对实现编程
  • 松耦合

策略模式

定义了算法族，分别封装起来，让它们之间可以互相替换，此模式让算法的变化独立于使用算法的客户。

示例：鸭子的设计

  • 子类行为交给实现行为接口的行为类来调用。

观察者模式

在对象之间定义一对多的依赖，当一个对象改变状态，依赖它的对象都会收到通知，并自动更新。

  • 有多个观察者，不可以依赖特定的通知次序。
  • 努力实现 Observable 类
  • 可观察者不知道观察者的细节，只知道观察者实现了观察者接口。

示例：气象站

Java 内置的观察者模式

java.util 包含最基本的 Observer 接口与 Observable 接口。

（1）把对象变成观察者

实现 Observer 接口然后调用 addObserver() deleteObserver()

（2）观察者送出通知

继承 Observable 类，标记状态改变 setChange() ，调用 notifyObservers() 或 notifyObservers(Object arg)

（3）观察者接收通知

uodate(Observable o, Object arg) 可以推（push）数据 Object arg 给观察者

Observable 是一个类，将关键的方法保护起来。可以扩展 java.util.Observable 或自己实现一整套观察者模式。

JDK 中的观察者模式

  • JavaBeans
  • Swing 中的 JButton ActionListener actionPerformed
  • RMI

装饰者模式

动态地将责任附加到对象上。想要扩展功能，装饰者提供有别于继承的另一种选择。

  • 装饰者会导致设计中出现许多小对象。
  • 组合和委托可用于在运动时动态地加上新的行为。
  • 装饰者可以在被装饰者的行为前后加上自己的行为，甚至替换整个被装饰者的行为。
  • 装饰者和被装饰者都继承了共同的超类，来达到类型匹配。

示例：咖啡店

Java I/O

工厂模式

抽象工厂模式：提供给一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类。

工厂方法模式：定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法把实例化推迟到子类。

abstract Product factoryMethod(String type)

依赖倒置原则：避免依赖具体类型，尽量依赖抽象。

  • 简单工厂不是真正的设计模式，但可以将客户程序从具体类解耦。
  • 抽象工厂使用对象组合，对象的创建被实现在工厂接口所暴露出来的方法中。
  • 工厂方法使用类，通过继承来创造对象。（对象是类的实例）
  • 抽象工厂的具体工厂经常实现工厂方法来创建他们的产品。
  • 当不知道将来需要实例化哪些具体类时可以用工厂方法，继承为子类并实现工厂方法，实例化推迟到子类。

示例：披萨店

单件模式

确保一个类只有一个实例，并提供全局访问点。

  • 实现单件模式需要私有的构造器、一个静态方法和一个静态变量。
public class Singleton {
    private static Singleton uniqueInstance;
    // 其他的有用实例化变量
    private Singleton(){}
    // 同步化处理多线程
    public static synchronized Singleton getInstance() {
        if(uniqueInstance == null) {
            uniqueInstance = new Singleton();
        }
        return uniqueInstance;
    }
    // 其他的有用方法
}

改善性能

  • 同步 getInstance() 方法最直接，但会降低性能。
  • 急切实例化 直接使用静态初始化 private static Singleton uniqueInstance = new Singleton(); 并用 public 静态方法返回
  • 双重检查加锁
public class Singleton {
    private volatile static Singleton uniqueInstance; // volatile禁止指令重排 多个线程能正确处理 uniqueInstance 变量
    private Singleton() {}
    public static Singleton getInstance() {
        if (uniqueInstance == null) {
            synchronized (Singleton.class) {
                if (uniqueInstance == null) {
                    uniqueInstance = new Singleton();
                } // close if
            } // close synchronized
        } // close if
        return uniqueInstance;
    } // close getInstance() method
}

命令模式

将“请求”封装成对象，以便使用不同的请求，队列或者日志来参数化其他对象。命令模式也支持可撤销的操作。redo

  • 命令模式将发出请求的对象和执行请求的对象解耦，命令对象作为中间的桥梁。
  • 命令对象封装了接收者和一个或一组动作。
  • 调用者通过调用命令对象的 execute() 发出的请求，调用接收者的动作。
  • 调用者可以接受命令当作参数。
  • 空对象 NoCommand ，调用它的 execute() 方法，对象什么也不做。

示例：餐厅、遥控器

餐厅 命令模式
服务员 Invoker
快餐厨师 Receiver
orderUp() execute()
订单 Command
顾客 Client
takeOrder() setCommand()

Client -> Command (Invoker.setCommand()) -> Invoker -> Invoker.execute()

撤销操作

  • 要有一个记录状态的变量。
  • undo() 和 execute() 同级

宏命令

命令的一种简单的延申，允许调用多个命令。借助栈来实现撤销。

Command[] commands = {a, b, c, d};
InstanceCommand instanceCommand = new InstanceCommand(commands);
Invoker.setCommand(instanceCommand);

应用

命令可以将运算块打包并当作一般的对象来传送，于是运算块可以在创建许久之后被调用，也支持不同的线程。

所以可以应用于：日志安排，线程池，工作队列。

适配器模式

将一个类的接口，转换成客户期望的另一个接口。适配器让原本不兼容的类可以合作无间。

  1. 客户通过目标接口调用适配器的方法对适配器发出请求 request()
  2. 适配器使用被适配者接口把请求转换成被适配者的一个或多个调用接口 translatedRequest()
  3. 客户接到调用的结果，但未察觉这一切是适配器在起转换作用。

示例：插座适配器

  • 大多数的适配器模式都是让一个适配器包装一个被适配器
  • 新旧系统共存，使用双向适配器。
// Adapter 和 Client 都实现了 Target接口
// 类似装饰者模式 都继承了共同的超类，来达到类型匹配
public class Adapter implements Target {
    Adaptee adaptee;
    public Adapter(Adaptee adaptee) {
        this.adaptee = adaptee;
    }
    public void clientMethod() {
        adaptee.adapteeMethod();
    }
}

对象适配器 & 类适配器

  • 类适配器通过多重继承来扩展类，注意，Java 中不能使用。
  • 对象适配器通过组合，实现接口，将客户调用的方法委托给被适配者，如果被适配者的子类加入了新的行为，只需要组合子类对象即可。

应用

适配旧的枚举器 Enumeration 和新的迭代器 Iterator ，向上兼容使用旧的方法但要抛弃新的功能，向下兼容使用新的方法。

装饰者 & 适配器

  • 装饰者不改变接口，但加入责任。
  • 适配器将一个接口转成另一个接口。
  • 外观模式让接口更简单。
public class EnumerationIterator(Enumeration enum) {
    Enumeration enum;
    public EnumerationIterator(Enumeration enum) {
        this.enum = enum;
    }
    public boolean hasNext() {
        return enum.hasMoreElements();
    }
    public Object next() {
        return enum.nextElement();
    }
    // 旧代码不支持的方法只能抛出异常
    public void remove() {
        throw new UnsupportedOperationException();
    }
}

外观模式

提供了一个统一的接口，用来访问子系统中的一群接口。外观定义了一个高层接口，让子系统更容易使用。

  • 将一个或多个复杂的类隐藏，只暴露出一个干净美好的外观。
  • 和适配器模式的区别不是包装的多少，而是意图，适配器是改变接口，外观模式是简化接口，当然也可以调用复杂的子系统接口。

最少知识原则

最好只能调用：

  • 该对象本身
  • 被当做方法的参数而传递进来的对象
  • 此方法所创建或实例化的任何对象
  • 对象的任何组件
// 不调用 从另一个调用中返回对象的方法 相当于向另一个对象的子部分发出请求 增加了耦合
public float getTemp() {
    Thermometer thermometer = station.getThermometer();
    return thermometer.getTemperature();
    // 另一种写法
    thermometer = station.getThermometer().getTemperature();
}
 
public float getTemp() {
    return station.getTemperature();
}

模板方法模式

在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以在不改变算法结构的情况下，重新定义算法中的某些步骤。

  • 抽象类可以定义具体方法、抽象方法和钩子。
  • 钩子是一种方法，在抽象类中不做事或默认的事，子类可以选择性覆盖。
  • 防止子类改变模板方法中的算法，可以将模板方法声明成 final 。
  • 将决策权放在高层模块，以便决定如何以及何时调用低层模板。
  • 区别于策略模式使用组合，模板方法模式使用继承.。
  • 工厂方法是模板方法的一种特殊版本，将对象的创建推迟到子类。
  • 抽象类主导一切，拥有算法，保护算法，容易修改，便于复用但由子类提供完整的实现。

示例：咖啡与茶

  • 都需要泡水和倒入杯中
  • 冲泡的具体方式不同
  • 根据需求加入配料
abstract class AbstractClass {
    final void templateMehod() {
        primitiveOperation1();
        primitiveOperation2();
        concreteOperation();
        hook();
    }
    abstract void primitiveOperation1();
    abstract void primitvieOperation2();
    void hook() {} // 默认什么都不做
}

比较

模板方法——子类决定如何实现算法中的某些步骤。

策略——封装可互换的行为，然后使用委托来决定要采用哪一个行为。

工厂方法——有子类决定实例化哪个具体类。

应用：对象数组排序

  1. 数组的模板提供了算法
public static void sort(Object[] a) {
    Object aux[] = (Object[])a.clone(); // 创建一个数组的拷贝
    mergeSort(aux, a, 0, a.length, 0);
}
// 算法的访问权限很重要
private static void mergeSort(Object src[], Object dest[], int low, int high, int off) {
    for (int i=low; i<high; i++) {
        // 前大后小则互换 当前位和前一位比
        for (int j=i; j>low && (Comparable)dest[j-1].compareTo((Comparable)dest[j]) > 0; j--) {
            swap(dest, j, j-1);
        } // close inner for loop
    } // close outer for loop
    return ;
}
  1. 实现自己的 compareTo() 方法
  2. 鸭子类要实现 Comparable 接口

应用：Swing & Applet

  • Swing 中的 paint(Graphics graphics) 方法就是一个 hook method
  • Applet 是一个能够在网页上执行的小程序 通过子类实现钩子来提供行为。

迭代器模式

提供一种方法顺序访问一个聚合对象中的各个元素，而又不暴露其内部的表示。

  • 迭代器将遍历聚合的工作封装进一个对象中
  • 迭代器提供了通用接口，使用聚合新的项来遍历时使用多态。
  • 一个类尽量只分配一个责任
  • 遍历的任务放在迭代器上，而不是聚合上，这样简化了聚合的接口和实现，让责任各得其所

示例：不同菜单（ArrayList 与数组）的遍历，服务员工作的简化

组合模式

允许将对象组合成树形结构来表现 “整体/部分” 的层次结构。组合能让客户以一致的方式处理个别对象和对象组合。

  • 组合模式可以同时包含个别对象和组合对象
  • 组合模式允许客户对个别对象（叶结点）及组合对象（包含叶结点的结点）有相同的处理方法
  • 组件：组合结构内的任意对象，可以是组合，也可以是叶结点

叶结点和组合 都继承了共同的超类，来达到类型匹配。

状态模式

允许对象在内部状态发生改变时改变它的行为，对象看起来好像修改了它的类。

  • 状态决定了下一个状态或状态的流向
  • 共享状态需要把每个状态都指定到静态的实例变量中****
  • 和策略模式有相同的类图，但意图不一样，策略模式通常会用行为和算法配置 Context 类，而状态模式允许 Context 随着状态的改变而改变行为。
  • 使用状态模式通常会导致设计中类的数目大量增加
  • 最好共享状态类给多个 Context 实例，设置静态类

示例：糖果机

  1. 找出所有状态
  2. 创建实例变量来持有目前的状态
  3. 整合动作（接口），每个动作都会造成状态的转换
  4. 创建一个状态机（类），为每个动作创建相应的方法，对不同状态做出反应，由后面的设计可知 if 判断书顺序无关

新的设计：

  1. 定义一个 State 接口，在这个接口内，糖果机的每个动作都有一个对应的方法。
  2. 为机器中的每个状态实现状态类，这些类将负责在对应的状态下进行机器的行为。
  3. 将动作委托到状态类。

复合模式 (Model-View-Controller)

MVC 是复合模式，结合了观察者模式、策略模式和组合模式。

视图：用来呈现模型。通常直接从模型中取得它需要显示的状态和数据。

控制器：获取用户的输入并解读其对模型的意思。

模型：模型持有所有的数据、状态和程序逻辑。模型没有注意到视图和控制器，虽然它提供了操纵和检索状态的接口，并发送状态改变通知给观察者。

实例：MP3播放器

  • 模型利用观察者模式让控制器和视图更新状态，控制器和视图是观察者
  • 视图和控制器实现了策略模式，控制器是视图的行为，可以更换
  • 视图内部使用组合模式来管理窗口、按钮等组件
  • 适配器模式用来将新的模型适配成已有的视图和控制器
